1. 这个项目解决什么问题？
``` text

  这个项目解决的是‘ic知识问答不可靠，缺少引用，不能调用领域工具’的问题，通过agent+rag+ic工具链，把普通的大模型封装成一个面向集成电路学习和工程辅助的专业问答系统。
业务视⻆：
    对学⽣：可以⽤它查询 Verilog、时序分析、乘法器优化、ASIC/FPGA 等 IC 知识，并看到资料来源。
    对⼯程场景：可以辅助检查 Verilog 代码、⽣成 SDC 时序约束建议、检索内部 IC ⽂档。
    对⾯试展示：它不是⼀个简单 ChatGPT wrapper，⽽是⼀个具备领域路由、⼯具调⽤、检索增强、引⽤治理和评测闭环的垂类 Agent系统
```
2. 用户输入后，系统完整流程是什么？
``` text
完整流程如下：用户通过fastapi的chat或者chat/stream接口接入问题，系统就会进入langgraph的主链路，先由pre_tool_router进行判断问题类型，比如普通知识查询，verilog代码检查，时序约束，然后调用对应的工具。知识类问题会走llamaindex+chroma的rag检索，代码问题可以走规则型代码分析，时序问题可以生成sdc约束建议。最终回答由大模型基于工具结果生成，并且服务端会重写引用，只允许展示本轮真实检索到的source，降低模型的幻觉。项目还包括etl建库，ic定制分快，cross encoder重排和ragas评测脚本，形成从数据、检索、生成、评测的完整闭环。
```
3. RAG 部分怎么做？
``` text
这个项目的 RAG 部分可以按 建库阶段 和 查询阶段 两条线讲。

  1. 建库阶段：PDF -> Chunk -> Embedding -> Chroma

  原始资料放在 data/ 目录，比如：

  data/Verilog 硬件描述语言.pdf
  data/verilog教程.pdf

  建库逻辑主要在 app/core/rag/retriever.py:97 和 scripts/build_knowledge.py：

  PDF
    -> PyPDFLoader / pypdf 抽文本
    -> ICCustomTextSplitter 做 IC 定制分块
    -> HuggingFaceEmbedding 生成向量
    -> LlamaIndex VectorStoreIndex
    -> Chroma 持久化到 chroma_db/

  这里的重点是 ICCustomTextSplitter，在 app/etl/ic_text_splitter.py:13。它不是简单按长度切，而是优先保留：

  - Verilog module ... endmodule 代码块
  - 章节边界
  - 时序图 / 波形图相关段落
  - 普通长文本再退回递归切分

  这样做是为了避免把一个 Verilog 模块或一个完整知识点切碎，影响后续检索质量。

  2. 查询阶段：用户问题 -> 检索 -> 重排 -> 返回证据

  用户请求进入 /api/v1/chat 后，LangGraphICAgent 会先在 app/core/agent/langgraph_agent.py:145 的 pre_tool_router 判断是不是 IC 领域问题。

  如果是知识类问题，就选择工具：

  ic_rag_search

  这个工具在 app/core/tools/builtin/ic_tools.py:213。它会：

  用户问题
    -> 轻量 query expansion
    -> ICRAGRetriever.retrieve()
    -> dense 检索候选结果
    -> CrossEncoder reranker 重排
    -> 结构化返回 content/source/page/score/chunk_id

  比如用户问“乘法器时序优化有哪些方法？”，工具会先扩展成类似：

  乘法器 乘法单元 booth wallace 阵列乘法器 关键路径 时序优化

  这样可以提高召回率。

  3. 检索器内部怎么做

  核心类是 app/core/rag/retriever.py:97。

  它做几件事：

  - 检查 data/ 里的 PDF 和 chroma_db 里的 source 是否一致
  - 如果 Chroma 不存在或 source 不一致，就自动重建索引
  - 使用 HuggingFaceEmbedding，默认配置是 bge-m3
  - 使用 VectorStoreIndex.as_retriever(similarity_top_k=...) 做向量检索
  - 使用 app/core/rag/reranker.py:12 做 CrossEncoder 重排
  - 最后返回 ICRetrievalResult

  结果结构是：

  {
    "content": "检索片段",
    "source": "Verilog 硬件描述语言.pdf",
    "page": "第12页",
    "score": 0.83,
    "chunk_id": "Verilog 硬件描述语言.pdf#p12#c3"
  }

  4. 答案生成和引用控制

  RAG 检索结果不会直接返回给用户，而是被注入到 answer_generator 的 system context 里，让模型严格基于片段回答。

  然后 app/core/rag/citation_rewriter.py:31 会做引用重写：

  - 删除模型自己生成的“参考资料”
  - 检查正文中的 source/page 是否真的来自本轮检索
  - 最后由服务端统一生成：

  参考资料（服务端生成）
  1. Verilog 硬件描述语言.pdf | 第12页

  这个是项目 RAG 的亮点：不完全相信模型写的引用，而是服务端根据真实检索结果重写。

  一句话总结：

  > 这个项目的 RAG 是基于 LlamaIndex + Chroma + bge-m3 的本地 PDF 检索链路，配合 IC 定制分块、query expansion、CrossEncoder 重排和服务端引用重写，实现了从专业
  > 文档召回到低幻觉回答的完整闭环。
```
4. Agent 部分怎么做？
```
Agent 部分核心在 app/core/agent/langgraph_agent.py:70，当前 /api/v1/chat 主链路用的是 LangGraphICAgent。

  它不是让大模型自由决定一切，而是做了一个比较可控的状态图：

  START
    -> pre_tool_router
        -> clarify
        -> tool_executor
        -> answer_generator
    -> END

  1. 入口

  用户请求先到 app/api/routes/chat.py:50，里面构造 Agent：

  agent = LangGraphICAgent(
      model_router=_build_router(),
      tool_registry=build_ic_tool_registry(),
  )

  然后调用：

  result = await agent.run(messages=messages, ...)

  2. pre_tool_router：前置路由

  pre_tool_router 负责判断用户问题该怎么处理。

  它会判断几类情况：

  - 空问题：返回澄清
  - 过短问题，比如“乘法器”“时序”：返回澄清
  - 非 IC 领域问题：不调用工具，后面严格拒答
  - IC 知识问题：选择 ic_rag_search
  - Verilog 代码问题：选择 ic_rag_search + verilog_code_analyzer
  - 时序/SDC 问题：选择 ic_rag_search + timing_constraint_suggester

  也就是说它像一个“调度器”，先决定该不该查资料、该调用哪些工具。

  3. tool_executor：执行工具

  如果 pre_tool_router 选中了工具，就进入 tool_executor。

  工具来自 app/core/tools/factory.py:9，实际注册在 app/core/tools/builtin/ic_tools.py:213：

  - ic_rag_search：查 IC 文档知识库
  - verilog_code_analyzer：做基础 Verilog 静态审查
  - timing_constraint_suggester：生成 SDC 约束建议

  工具执行结果会被记录成 tool_events，返回给接口，前端也可以展示工具调用过程。

  4. answer_generator：生成最终回答

  工具执行完后，answer_generator 把工具结果拼进 system prompt，然后调用模型生成最终回答。

  这里的策略比较严格：

  - 必须基于 ic_rag_search 的检索片段回答
  - 不允许模型脱离工具结果自由补答
  - 不让模型自己写参考资料区块
  - 如果 RAG 没命中，就走严格拒答

  严格拒答大概是：

  【严格拒答】
  当前知识库未命中可引用证据，无法基于资料直接回答。
  请补充更具体关键词，或更新/上传相关 PDF 后重试。

  5. citation_rewriter：服务端引用治理

  Agent 生成答案后，接口层还会调用 app/core/rag/citation_rewriter.py:31。

  它的作用是：

  - 删除模型自己编的参考资料
  - 检查正文中的引用是否真实来自本轮检索
  - 由服务端重新生成可信引用

  所以最终不是“模型说引用了什么就信什么”，而是后端用真实 sources 重写。

  6. 结果返回

  最后返回：

  ChatResponse(
      answer=...,
      trace_id=...,
      sources=...,
      tool_events=...
  )

  所以用户不仅能看到答案，还能看到：

  - 用了哪个模型
  - 调用了哪些工具
  - 检索到了哪些来源
  - trace_id 是多少

  一句话总结：

  > Agent 部分用 LangGraph 做了一个可控的三段式流程：先路由判断问题类型，再调用 IC 领域工具，最后基于工具结果生成答案；同时加入短问题澄清、非领域拒答、RAG 未命
  > 中严格拒答和服务端引用治理，避免模型自由发挥导致幻觉。
```

用户请求从哪里进来？
``` text
用户请求从 FastAPI 入口进来，FastAPI chat endpoint 接收 ChatRequest，主要进到真正的用户业务请求主要进 app/api/routes/chat.py:50：
      - 非流式：POST /api/v1/chat
      - 流式：POST /api/v1/chat/stream
```
经过哪些模块？
``` text
首先由api路由层（关键⽂件：app/api/routes/chat.py）接受请求，之后（LangGraphICAgent 编排层）根据不同的问题类型选择不同的工具，再经过ic工具层（关键⽂件：app/core/tools/builtin/ic_tools.py）执行具体的工具逻辑，在经过rag检索层（关键⽂件：app/core/rag/retrieve.py），判断回答是否可信，在经过. Retriever 从 Chroma / LlamaIndex 检索⽂档，必要时 rerank，最后由生成层生成回答
```
每个模块负责什么？
``` text
更具体一点：

  1. app/main.py:26
     创建 FastAPI 应用，挂载 chat.router，所以 /api/v1/chat 才能被访问。
```

出了错去哪查？
``` text
查顺序建议这样：
  1. 接口返回
      - /api/v1/chat 出错会返回 HTTP 500 或 503，看返回里的 detail。
      - 正常响应里有 trace_id，先记下来。
      - tool_events 里能看到工具是否成功，字段 ok=false 时看 result。
  2. 聊天接口问题
      - 查 app/api/routes/chat.py:50
      - 关键日志：
          - chat 失败
          - chat_stream 失败
          - LangGraph 路由完成 trace=... tools=... reason=...
  3. Agent 路由/工具调用问题
      - 查 app/core/agent/langgraph_agent.py:70
      - 重点看：
          - 是否进入 needs_clarification
          - selected_tools 选了什么
          - tool_outputs 里哪个工具失败
          - 是否走了 strict_refusal
  4. RAG 检索问题
      - 查 app/core/rag/retriever.py:97
      - 常见日志：
          - 未发现可用索引，开始首次构建
          - 检测到知识库 source 不一致
          - CrossEncoder 重排序失败
      - 也可以单独跑：
        python evaluation/retrieval_smoke_test.py --query "乘法器时序优化有哪些方法？" --top-k 3
  5. 模型调用问题
      - 查 app/infrastructure/llm/model_router.py:41
      - 常见原因是：
          - OPENAI_API_KEY 不对
          - OPENAI_API_BASE 不对
          - 模型名不对
          - API 超时/限流
      - 配置在 app/config.py:9
  6. 文档上传/入库问题
      - 查 app/api/routes/document.py:56
      - 常见日志：
          - 保存上传文件失败
          - ETL 失败
          - 向量入库失败
          - 查询文档列表失败
  7. 数据库问题
      - 先访问：

        curl http://127.0.0.1:8000/api/v1/health/ready

      - 如果返回 degraded，看 PostgreSQL 是否启动、DATABASE_URL 是否正确。
      - DB 初始化在 app/infrastructure/database/session.py:36
  补充一点：项目有 Tracer，但当前没有暴露“按 trace_id 查询”的 HTTP 接口，所以实际排查主要还是看终端日志、接口响应里的 trace_id/tool_events/sources，再顺着对应
  模块查。
```
